## 1. Проблемы и сигналы — накопительный трекер

### Суть изменения
Добавить блок «Проблемы и сигналы» в **левую панель карточки руководителя** (профиль).
Блок накапливает проблемы из всех встреч, отслеживает их статус и автоматически
обновляется при каждом AI-анализе расшифровки.

### Где располагается
Левая панель карточки руководителя — после блока «Слабые стороны», перед комментариями консультанта.

### Визуальная структура блока

```
⚠️ Проблемы и сигналы                              [+ Добавить]

• Согласование решений с коммерцией        [активна · 3 встречи]  [✕]
• Размытость зон ответственности           [активна · 2 встречи]  [✕]
• Динамика с Олегом на общих совещаниях    [активна · 1 встреча]  [✕]
~~Нет структуры в еженедельных планёрках~~ [решена · встреча №4]
```

Активные — обычный текст + кнопка закрытия `[✕]`.
Решённые — зачёркнутый текст + номер встречи, где закрылась.

### Ручное управление

- Кнопка **«+ Добавить»** — добавить проблему вручную (без привязки к встрече).
  `first_seen_meeting_id = NULL` для ручных записей.
- Кнопка **«✕»** рядом с активной проблемой — закрыть вручную (без привязки к встрече).
  `resolved_meeting_id = NULL` для ручного закрытия.
- Решённые проблемы отображаются зачёркнутыми, скрываются через 30 дней после закрытия.

---

### Схема данных

Новая таблица `manager_problems` в Supabase:

```sql
CREATE TABLE manager_problems (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  manager_id            UUID REFERENCES managers(id) ON DELETE CASCADE,
  text                  TEXT NOT NULL,
  status                TEXT DEFAULT 'active',  -- 'active' | 'resolved'
  source                TEXT DEFAULT 'ai',       -- 'ai' | 'manual'
  first_seen_meeting_id UUID REFERENCES meetings(id) ON DELETE SET NULL,
  resolved_meeting_id   UUID REFERENCES meetings(id) ON DELETE SET NULL,
  meeting_count         INTEGER DEFAULT 1,       -- сколько встреч проблема активна
  created_at            TIMESTAMPTZ DEFAULT now(),
  updated_at            TIMESTAMPTZ DEFAULT now()
);
```

RLS: те же политики что у `managers` — доступ только аутентифицированным пользователям.

### Механика автообновления

При AI-анализе расшифровки (промпт 2) добавить в `diagnostic_extension` новый блок:

```json
"problems_delta": [
  {
    "problem_id": "abc123",
    "action": "ongoing"
  },
  {
    "problem_id": "def456",
    "action": "resolved",
    "resolution_note": "Внедрена повестка, подтверждено на встрече"
  },
  {
    "text": "Размытость зон ответственности",
    "action": "new"
  }
]
```

**Возможные значения `action`:**
- `new` — проблема появилась впервые. Поле `text` обязательно, `problem_id` отсутствует. Создать новую запись в `manager_problems`.
- `ongoing` — проблема была раньше и продолжается. Поле `problem_id` обязательно. Увеличить `meeting_count` на 1.
- `resolved` — проблема закрылась. Поле `problem_id` обязательно. Поставить `status = 'resolved'`, записать `resolved_meeting_id`.

**Важно:** для `ongoing` и `resolved` AI получает список активных проблем с их `id` и сопоставляет явно — без нечёткого текстового матчинга.

### Дополнение к промпту 2 (анализ расшифровки)

Добавить в **system**-промпт:

```
Дополнительно верни "problems_delta" — массив изменений по проблемам и сигналам.

Список активных проблем этого руководителя (используй эти id для ongoing/resolved):
[
  { "id": "abc123", "text": "Согласование решений с коммерцией" },
  { "id": "def456", "text": "Нет структуры в еженедельных планёрках" }
]

Для каждой проблемы из расшифровки укажи action:
  "new"      — проблема не встречалась ранее (problem_id не нужен, укажи text)
  "ongoing"  — проблема из списка выше продолжается (укажи problem_id из списка)
  "resolved" — есть явные признаки закрытия (укажи problem_id из списка + resolution_note)

Если активных проблем нет — верни пустой массив [].
```

Список активных проблем формируется на сервере перед запросом:
`SELECT id, text FROM manager_problems WHERE manager_id = ? AND status = 'active'`

## 2. Telegram-уведомление накануне встречи

### Суть
За день до встречи (накануне) Telegram-бот отправляет уведомление с активными проблемами
руководителя. Пользователь закрывает решившиеся проблемы нажатием inline-кнопок.

### Текст уведомления (шаблон)

```
📅 Встреча №{N} с {Имя руководителя} — {дата}

Активные проблемы из предыдущих встреч:
1. Согласование решений с коммерцией (3 встречи)
2. Размытость зон ответственности (2 встречи)
3. Динамика с Олегом на общих совещаниях (1 встреча)

Если что-то из этого уже решилось — нажмите кнопку:

[✅ Закрыть №1]
[✅ Закрыть №2]
[✅ Закрыть №3]
```

Если активных проблем нет — более краткое уведомление:

```
📅 Встреча №{N} с {Имя руководителя} — {дата}

Активных проблем нет. Удачной встречи!
```

### Механика обработки нажатия кнопки

1. Пользователь нажимает `[✅ Закрыть №1]`
2. Telegram присылает `callback_query` на `/api/telegram/webhook`
3. В `callback_data` кнопки хранится `problem_id` (не порядковый номер)
4. Сервер помечает проблему как `resolved` (`resolved_meeting_id = NULL` — закрыто вручную)
5. Telegram получает `answerCallbackQuery` с текстом: «Проблема закрыта в карточке {Имя}.»
6. Кнопка в сообщении обновляется: `[✅ Закрыть №1]` → `[☑️ Закрыта]` (через `editMessageReplyMarkup`)

### Триггер

**Vercel Cron Job** — запускается ежедневно в 9:00 утра.

```json
// vercel.json
{
  "crons": [
    {
      "path": "/api/cron/meeting-reminders",
      "schedule": "0 6 * * *"
    }
  ]
}
```

*(6:00 UTC = 9:00 MSK)*

Логика cron-эндпоинта:
```
SELECT meetings JOIN managers
WHERE meetings.date = CURRENT_DATE + 1
  AND meetings.status = 'preparation'
  AND managers.telegram_chat_id IS NOT NULL (из ai_settings)
```

Для каждой найденной встречи — отправить уведомление с активными проблемами.

### Telegram Webhook

Зарегистрировать один раз при деплое:
```
POST https://api.telegram.org/bot{TOKEN}/setWebhook
body: { "url": "https://your-app.vercel.app/api/telegram/webhook" }
```

Endpoint `/api/telegram/webhook` обрабатывает:
- `callback_query` — нажатие inline-кнопок (закрытие проблем)
- (расширяемо на другие типы update в будущем)

### Настройка в /settings

Добавить в блок Telegram-настроек:
- Тогл **«Уведомление накануне встречи»** вкл/выкл (по умолчанию: вкл)
- Поле с тоглом хранится в `ai_settings.meeting_reminder_enabled` (новое поле `BOOLEAN DEFAULT true`)

**Поле «за сколько часов» — не нужно.** Уведомление всегда накануне, без настройки времени.

## 3. Остальные улучшения (приоритизированы)

### 3.1 Поиск — высокий приоритет, низкая сложность

Глобальный поиск в шапке приложения.
Ищет по: имени руководителя, названию компании, тексту встречи, проблемам.

**Реализация:** Supabase full-text search (`to_tsvector` / `to_tsquery`).
Для `transcription_text` — отдельный `tsvector`-столбец с `GIN`-индексом (текст большой).

### 3.2 Разные модели для разных задач — средний приоритет, низкая сложность

В `/settings` добавить три отдельных дропдауна вместо одного:
- Модель для генерации сценария
- Модель для анализа расшифровки
- Модель для чата с AI

Каждый дропдаун — список актуальных моделей с OpenRouter.

**Миграция:** поле `preferred_model` в `ai_settings` → три поля:
`scenario_model`, `analysis_model`, `chat_model`.
Дефолт для всех — текущее значение `preferred_model`.

**Мотивация:** сценарий и анализ можно делать на более дешёвой модели,
чат — на более мощной (Claude Opus и др.).

### 3.3 Откат трека — средний приоритет, низкая сложность

Рядом с кнопкой «Показать diff» добавить кнопку **«Восстановить эту версию»**.
Версии уже хранятся в `document_versions` — нужно только добавить кнопку restore
(записать старое `content` как новую версию текущего документа).

### 3.4 Уведомления о предстоящих встречах без проблем — низкий приоритет

Решается пунктом 2: если активных проблем нет — отправляется краткое напоминание.
Отдельной задачи не требует.

### 3.5 Связь целей с договорённостями — низкий приоритет, средняя сложность

При создании договорённости в карточке встречи — необязательный дропдаун
«Связать с целью» (список целей из вкладки «Цели» этого руководителя).
Прогресс цели обновляется при выполнении связанных договорённостей.

**Зависимость:** договорённости сейчас живут в `diagnostic_extension` (JSON).
Для полноценной связи нужна отдельная таблица `commitments`. Это отдельная задача.

### 3.6 Интеграция с Plaud — зависит от их API

Проверить наличие Plaud Webhook API или Export API.
Если есть — автозагрузка расшифровок по имени файла или по meeting ID.
Приоритет поставить после проверки документации.

## 4. Что НЕ меняется

- Количество вкладок в карточке руководителя остаётся прежним (7 вкладок)
- Проблемы и сигналы идут в **левую панель** (профиль), не в новую вкладку
- Поле `date` в таблице `meetings` **не меняется** (остаётся `DATE`, без времени)
- Существующие промпты меняются только в части добавления `problems_delta`
- Матрица прав доступа не меняется
