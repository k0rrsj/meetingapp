## 1. Замкнутый цикл — алгоритм работы

Замкнутый цикл — **ключевая бизнес-логика системы**. Реализуется в момент создания новой карточки встречи.

### 1.1 Триггер

Саша нажимает «Создать встречу» на карточке руководителя.

### 1.2 Алгоритм (серверная логика, Next.js API Route)

```
POST /api/managers/:managerId/meetings
```

**Шаг 1 — Поиск предыдущей встречи**

```sql
SELECT * FROM meetings
WHERE manager_id = :managerId
ORDER BY date DESC, meeting_number DESC
LIMIT 1;
```

- Если встреча найдена и её статус = `closed` → использовать для формирования контекста (основной путь)
- Если встреча найдена, но статус ≠ `closed` → использовать, но добавить флаг `context_from_unclosed: true` (предупреждение в UI)
- Если встреч нет → `previous_context = null`, новая встреча создаётся без контекста

**Шаг 2 — Сборка контекста**

Из предыдущей встречи извлекаются следующие поля:

```typescript
interface PreviousContext {
  meetingNumber: number;
  date: string;
  conclusions: string;
  actionPlan: string;
  problemsSignals: string;
  consultantComments: Comment[]; // комментарии Даниила к той встрече
  managerProfileComments: Comment[]; // актуальные комментарии к профилю руководителя
}
```

**Шаг 3 — Формирование текста `previous_context`**

Контекст сохраняется в двух форматах:
- `previous_context_json` — структурированный JSON (для AI)
- `previous_context_text` — читаемый текст (для отображения в UI)

Шаблон текстового контекста:
```
Встреча №{N} от {дата}

ВЫВОДЫ:
{conclusions}

ПЛАН ДЕЙСТВИЙ:
{actionPlan}

ПРОБЛЕМЫ И СИГНАЛЫ:
{problemsSignals}

КОММЕНТАРИИ КОНСУЛЬТАНТА:
{consultantComments}
```

**Шаг 4 — AI-генерация (если включено)**

Если у пользователя настроен OpenRouter API Key и выбрана модель:
1. Отправить запрос к `/api/ai/generate-scenario` с контекстом
2. Получить сгенерированный сценарий
3. Записать в поле `scenario` новой встречи
4. Аналогично сгенерировать `transcription_prompt`

Если AI недоступен / ключ не настроен → поля `scenario` и `transcription_prompt` остаются пустыми, Саша заполняет вручную.

**Шаг 5 — Создание записи**

```typescript
const newMeeting = {
  manager_id: managerId,
  meeting_number: lastMeetingNumber + 1, // автоинкремент
  date: null, // заполняется Сашей
  type: 'one_to_one',
  status: 'preparation',
  previous_context_json: contextJson,
  previous_context_text: contextText,
  context_from_unclosed: boolean,
  scenario: aiGeneratedScenario ?? null,
  transcription_prompt: aiGeneratedPrompt ?? null,
  // остальные поля — null
};
```

### 1.3 Edge Cases

| Ситуация | Поведение системы |
|---------|-------------------|
| Первая встреча с руководителем | `previous_context = null`, карточка создаётся без контекста |
| Предыдущая встреча не закрыта | Контекст берётся, но показывается предупреждение: «Контекст взят из незакрытой встречи №N» |
| AI-генерация зависла > 30 сек | Timeout, поля остаются пустыми, уведомление пользователю |
| AI вернул пустой ответ | Поля остаются пустыми, уведомление пользователю |
| Несколько встреч в один день | Берётся с наибольшим `meeting_number` |

## 2. Жизненный цикл карточки встречи

### 2.1 Статусы и переходы

```
preparation → conducted → processed → closed
```

Переходы только вперёд. Откат статуса назад не разрешён.

### 2.2 Права на переход статусов

Оба пользователя могут менять статус (для удобства работы).

### 2.3 Доступность полей по статусам

| Поле | preparation | conducted | processed | closed |
|------|-------------|-----------|-----------|--------|
| Дата встречи | редактировать | редактировать | только чтение | только чтение |
| Контекст из предыдущих | только чтение | только чтение | только чтение | только чтение |
| Сценарий встречи | редактировать | только чтение | только чтение | только чтение |
| Промпт для расшифровки | редактировать | только чтение | только чтение | только чтение |
| Расшифровка (текст) | — | редактировать | только чтение | только чтение |
| Файл расшифровки | — | загружать | только чтение | только чтение |
| Ключевые факты | — | — | редактировать | только чтение |
| Проблемы и сигналы | — | — | редактировать | только чтение |
| Выводы | — | — | редактировать | только чтение |
| Сильные / слабые стороны | — | — | редактировать | только чтение |
| План действий | — | — | редактировать | только чтение |
| Сценарий следующей встречи | — | — | редактировать | только чтение |
| Комментарии | редактировать | редактировать | редактировать | редактировать |

### 2.4 Валидация перехода к статусу `closed`

Переход `processed → closed` блокируется, если не заполнены:
- `conclusions` (выводы)
- `action_plan` (план действий)

При попытке перейти без заполненных полей — показать ошибку: «Для закрытия встречи заполните: Выводы, План действий».

### 2.5 Автоматические действия при переходах

| Переход | Автоматическое действие |
|---------|------------------------|
| `preparation → conducted` | Устанавливается `conducted_at = now()` |
| `processed → closed` | Инициируется подготовка контекста для следующей встречи (фоновая задача не нужна — контекст собирается при создании следующей встречи) |

## 3. AI-интеграция (OpenRouter)

### 3.1 Архитектура

Все обращения к AI идут через сервер Next.js (API Routes), никогда не с клиента. Ключ OpenRouter хранится в переменных окружения сервера.

```
Client → Next.js API Route → OpenRouter API → LLM модель
```

### 3.2 Конфигурация модели

- Пользователь выбирает модель в настройках (сохраняется в таблице `ai_settings`)
- По умолчанию: `openai/gpt-4o`
- При каждом запросе к AI берётся актуальная модель из `ai_settings` для текущего пользователя
- Fallback при ошибке: не переключаться автоматически, уведомить пользователя

### 3.3 Системные промпты

#### Генерация сценария встречи

```
Системный промпт:
Ты — помощник управленческого консультанта. На основе контекста предыдущих встреч 
с руководителем подготовь сценарий для следующей встречи one-to-one.

Сценарий должен включать:
1. Ключевые темы для обсуждения (3-5 пунктов)
2. Открытые вопросы по каждой теме
3. Темы для проверки прогресса по предыдущим договорённостям

Формат: структурированный текст, без лишних вступлений.
Язык: русский.

Контекст предыдущих встреч:
{previous_context_text}

Профиль руководителя:
{manager_profile}
```

#### Генерация промпта для расшифровки

```
Системный промпт:
На основе сценария встречи сформируй промпт для AI-расшифровки записи встречи.
Промпт должен попросить AI извлечь:
- Ключевые факты и решения
- Проблемы и сигналы
- Договорённости и следующие шаги
- Оценку сильных и слабых сторон руководителя

Сценарий встречи:
{scenario}
```

#### Генерация контекста из нескольких встреч

```
Системный промпт:
Сожми информацию о {N} встречах с руководителем в краткий контекст (не более 500 слов).
Выдели: ключевые тренды, незакрытые вопросы, прогресс по договорённостям.

Встречи (от старой к новой):
{meetings_summary}
```

### 3.4 Формат запросов к OpenRouter

```typescript
const response = await fetch('https://openrouter.ai/api/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${process.env.OPENROUTER_API_KEY}`,
    'Content-Type': 'application/json',
    'HTTP-Referer': process.env.NEXT_PUBLIC_APP_URL,
    'X-Title': 'Meeting Intelligence App',
  },
  body: JSON.stringify({
    model: userSettings.preferred_model,
    messages: [
      { role: 'system', content: systemPrompt },
      { role: 'user', content: userMessage },
    ],
    max_tokens: 2000,
    temperature: 0.7,
  }),
});
```

### 3.5 Обработка ошибок AI

| Ошибка | Поведение |
|--------|-----------|
| 401 Unauthorized | «Проверьте API ключ OpenRouter в настройках» |
| 429 Rate Limit | «Превышен лимит запросов. Попробуйте через минуту» |
| 500 Server Error | «Сервис AI временно недоступен. Попробуйте позже» |
| Timeout (> 30 сек) | «Генерация заняла слишком много времени. Попробуйте снова» |
| Пустой ответ | «AI не вернул результат. Попробуйте снова или заполните вручную» |

### 3.6 Примерная стоимость токенов

| Операция | Входящие токены | Исходящие токены | Итого (GPT-4o) |
|---------|----------------|-----------------|----------------|
| Генерация сценария | ~500 | ~600 | ~$0.006 |
| Генерация промпта | ~300 | ~400 | ~$0.004 |
| Сборка контекста | ~1000 | ~500 | ~$0.009 |
| **На один цикл встречи** | | | **~$0.02** |

## 4. Загрузка файлов расшифровок

### 4.1 Поддерживаемые форматы

- `.txt` — читается напрямую
- `.docx` — текст извлекается через библиотеку `mammoth`

### 4.2 Процесс загрузки

```
Client (выбор файла)
  → POST /api/files/upload (multipart/form-data)
  → Валидация (формат, размер ≤ 10 МБ)
  → Загрузка в Supabase Storage (bucket: transcriptions)
  → Извлечение текста (txt / docx)
  → Сохранение: meeting.transcription_file_url = storage URL
  → Сохранение: meeting.transcription_text = извлечённый текст
  → Ответ клиенту: { fileUrl, textPreview }
```

### 4.3 Структура хранения в Supabase Storage

```
bucket: transcriptions (private)
  /{managerId}/{meetingId}/transcription.{ext}
```

### 4.4 Edge Cases

| Ситуация | Поведение |
|---------|-----------|
| Файл > 10 МБ | Ошибка до начала загрузки: «Файл слишком большой (макс. 10 МБ)» |
| Неподдерживаемый формат | Ошибка: «Поддерживаются только .txt и .docx» |
| Повреждённый .docx | Ошибка: «Не удалось прочитать файл. Попробуйте экспортировать в .txt» |
| Повторная загрузка | Старый файл перезаписывается, текст обновляется |

## 5. Комментарии

### 5.1 Полиморфная привязка

Комментарии привязываются к двум типам сущностей:

```typescript
type CommentTarget = 'manager' | 'meeting';

interface Comment {
  id: string;
  target_type: CommentTarget;
  target_id: string;      // manager.id или meeting.id
  user_id: string;
  user_role: 'consultant' | 'assistant';
  text: string;
  created_at: string;
}
```

### 5.2 Роль комментариев в замкнутом цикле

При формировании `previous_context` для новой встречи включаются:
1. Все комментарии Даниила (роль `consultant`) к **последней встрече**
2. Последние 3 комментария Даниила к **профилю руководителя**

Комментарии ассистента в контекст **не включаются**.

### 5.3 Отображение

- Комментарии отображаются в хронологическом порядке
- Каждый комментарий: аватар/имя автора, дата, текст
- Нет тредов, нет реакций — только линейный список
- Редактирование и удаление: автор может редактировать/удалять свои комментарии

## 6. Авторизация и роли

### 6.1 Механизм

- Supabase Auth (email/password)
- После входа: JWT-токен в httpOnly cookie
- Middleware Next.js проверяет токен на каждый запрос к защищённым маршрутам
- Роль пользователя хранится в таблице `user_profiles` и в Supabase Auth metadata

### 6.2 Создание пользователей

Два фиксированных пользователя создаются вручную через Supabase Dashboard:
- `daniil@example.com` → роль `consultant`
- `sasha@example.com` → роль `assistant`

Регистрация через UI не предусмотрена.

### 6.3 Проверка ролей

```typescript
// middleware.ts
export async function middleware(request: NextRequest) {
  const session = await getSession(request);
  if (!session) return redirectToLogin(request);

  const role = session.user.user_metadata.role;

  // Защита маршрутов только для assistant
  if (request.nextUrl.pathname.startsWith('/api/meetings') &&
      request.method === 'POST' &&
      role !== 'assistant') {
    return new Response('Forbidden', { status: 403 });
  }
}
```

### 6.4 Supabase RLS политики

```sql
-- Все аутентифицированные пользователи читают все данные
CREATE POLICY "authenticated_read_all" ON companies
  FOR SELECT TO authenticated USING (true);

-- Только assistant создаёт встречи
CREATE POLICY "assistant_insert_meetings" ON meetings
  FOR INSERT TO authenticated
  USING (auth.jwt() ->> 'role' = 'assistant');

-- Комментарии: читают все, пишут все, удаляют/редактируют только автор
CREATE POLICY "comments_author_update" ON comments
  FOR UPDATE TO authenticated
  USING (user_id = auth.uid());
```

## 7. Агрегированные метрики

### 7.1 Карточка компании

```typescript
interface CompanyCardMetrics {
  activeManagersCount: number;    // managers WHERE status = 'active'
  lastMeetingDate: Date | null;   // MAX(meetings.date) по всем менеджерам
  overallProgress: number;        // % закрытых встреч от общего числа
  status: 'active' | 'completed';

Считается через SQL-запрос при загрузке списка компаний (не кешируется в MVP).

interface ManagerCardMetrics {
  meetingsCount: number;          // COUNT(meetings)

}
```

## 8. Автосохранение полей карточки встречи

### 8.1 Стратегия

- Все текстовые поля карточки встречи сохраняются с дебаунсом **1500ms** после последнего изменения
- При уходе со страницы — принудительное сохранение
- Индикатор состояния: «Сохранено» / «Сохранение...» / «Ошибка сохранения»

### 8.2 Исключения

- Статус встречи меняется только явным нажатием кнопки
- Загрузка файла — отдельное действие, не автосохранение

---

## 9. Нумерация встреч

- `meeting_number` — порядковый номер встречи с конкретным руководителем
- Рассчитывается на сервере: `MAX(meeting_number) + 1` для данного `manager_id`
- Нумерация не зависит от типа встречи
- Не редактируется пользователем
