## 1. Проблема и цель продукта

### Проблема

Управленческий консультант Даниил ведёт регулярные встречи (one-to-one) с руководителями клиентских компаний. Каждый цикл работы состоит из:

- подготовки сценария встречи (вручную, на основе предыдущих разговоров)
- проведения встречи и записи в Plaud
- получения расшифровки и её анализа
- формирования плана действий и сценария следующей встречи

Сейчас всё это делается вручную через Word, Excel, чаты с ИИ и интерфейс Plaud. Ключевая проблема — **каждый раз нужно вручную вспоминать и копировать контекст** из предыдущих встреч. Это занимает ~30% рабочего времени.

### Цель продукта

Создать веб-приложение, которое автоматизирует замкнутый цикл работы со встречами: **выход каждой встречи автоматически становится входом следующей.**

### Метрики успеха

| Метрика | Цель |
|---------|------|
| Сокращение ручной работы Даниила | ≥ 30% |
| Время подготовки к встрече | < 5 минут вместо 15-20 |
| Количество ручных операций на цикл встречи | сокращение с ~12 до ~4 |

## 2. Пользователи

### Даниил — управленческий консультант (роль: `consultant`)

**Задачи:**
- Перед встречей: открыть карточку руководителя, прочитать контекст и сценарий
- После встречи: просмотреть результаты анализа, добавить комментарии
- Отслеживать прогресс по каждому руководителю и компании

**Частота использования:** 2-3 раза в день (перед встречей + после)

**Ключевое требование:** максимально быстрый доступ к нужной информации, минимум лишних действий

### Саша — ассистент (роль: `assistant`)

**Задачи:**
- Создавать и заполнять карточки встреч
- Вносить расшифровки из Plaud (загрузка файла)
- Заполнять анализ: ключевые факты, проблемы, выводы, план действий

**Частота использования:** 1-2 раза на каждую встречу (подготовка + постобработка)

## 3. Объём системы (Scope)

### MVP — только в скоуп

- Тип встреч: **только one-to-one / менторинг**
- Количество компаний: 3
- Количество руководителей: ~15 (по 5 на компанию)
- Пользователи: 2 (Даниил + Саша)
- Расшифровки: загрузка файлов (txt, docx), позже — API Plaud
- AI: интеграция с OpenRouter с переключением моделей

### Вне MVP (отложено)

- Тип встреч «Диагностика»
- Мобильное приложение
- Интеграция с Plaud по API
- Многопользовательский режим (более 2 пользователей)
- Экспорт в PDF/Word
- Уведомления и напоминания

## 4. Функциональные требования

### FR-1: Замкнутый цикл (ключевая функция)

При создании новой встречи для руководителя система **автоматически** формирует блок «Контекст из предыдущих встреч» на основе последней закрытой встречи с этим руководителем, включая выводы, план действий, проблемы и комментарии Даниила.

### FR-2: Трёхуровневая навигация

Приложение организовано в три уровня:
1. **Компании** — список всех компаний с агрегированными метриками
2. **Руководители** — список руководителей выбранной компании
3. **Карточка руководителя** — профиль + история встреч

### FR-3: Карточка встречи

Каждая встреча — структурированная карточка с фиксированным набором полей (см. раздел 6). Карточка имеет статус, который последовательно меняется: `подготовка → проведена → обработана → закрыта`.

### FR-4: Профиль руководителя

Статичный блок с информацией о руководителе, которая заполняется один раз:
- Имя, должность, роль в команде
- Контекст (кто такой, особенности)
- Запрос от директора
- Сильные и слабые стороны
- Комментарии Даниила (редактируемое поле)

### FR-5: Комментарии

Оба пользователя могут оставлять комментарии на двух уровнях:
- В профиле руководителя (глобальные наблюдения)
- Внутри карточки встречи (по конкретной встрече)

Комментарии видны обоим и учитываются при формировании контекста следующей встречи.

### FR-6: Загрузка расшифровок

Саша загружает файл расшифровки из Plaud (форматы: txt, docx). Текст автоматически извлекается и сохраняется в поле `transcription_text` карточки встречи.

### FR-7: AI-интеграция (OpenRouter)

Система предоставляет AI-помощь для:
- Генерации контекста из предыдущих встреч
- Генерации сценария следующей встречи
- Генерации промпта для расшифровки

Используемая модель выбирается в настройках и может быть изменена.

### FR-8: Авторизация с ролями

Два фиксированных пользователя с разными правами (см. раздел 7). Вход через email/password.

### FR-9: Визуализация прогресса

На карточках компаний и руководителей отображается прогресс: количество встреч, дата последней, статус работы.

## 5. Нефункциональные требования

| Требование | Параметр |
|-----------|---------|
| Время отклика UI | < 1 секунды для навигации |
| Время AI-генерации | < 15 секунд (с индикатором загрузки) |
| Доступность | Веб-браузер, desktop-first, базовая мобильная адаптивность |
| Безопасность | HTTPS, авторизация обязательна, данные изолированы от публичного доступа |
| Надёжность | Данные хранятся в Supabase (managed PostgreSQL), резервное копирование автоматическое |
| Масштабируемость | Не критична для MVP (≤2 пользователей, ≤500 записей) |

## 6. User Stories и Acceptance Criteria

### US-1: Подготовка к встрече (Даниил)

**Как** Даниил, **я хочу** открыть карточку руководителя перед встречей **и увидеть** сценарий и контекст из предыдущих встреч, **чтобы** быть готовым без дополнительных поисков.

**Acceptance Criteria:**
- [ ] Карточка руководителя открывается за < 1 секунды
- [ ] Блок «Контекст из предыдущих встреч» содержит выводы и план действий из последней закрытой встречи
- [ ] Сценарий встречи виден на карточке встречи со статусом «подготовка»
- [ ] Если предыдущих встреч нет — отображается пустой блок с подсказкой

---

### US-2: Автоматический контекст при создании встречи (Саша)

**Как** Саша, **я хочу** создать новую карточку встречи **и получить** автоматически заполненный контекст из предыдущей, **чтобы** не копировать данные вручную.

**Acceptance Criteria:**
- [ ] При создании новой встречи поле «Контекст из предыдущих встреч» заполняется автоматически
- [ ] Контекст берётся из последней встречи со статусом «закрыта» для этого руководителя
- [ ] Контекст включает: выводы, план действий, проблемы и сигналы, комментарии Даниила
- [ ] Если предыдущая встреча не закрыта — берётся последняя по дате независимо от статуса, с предупреждением
- [ ] Если встреч не было — поле остаётся пустым

---

### US-3: Загрузка расшифровки (Саша)

**Как** Саша, **я хочу** загрузить файл расшифровки из Plaud, **чтобы** текст автоматически появился в карточке встречи.

**Acceptance Criteria:**
- [ ] Поддерживаются форматы: txt, docx
- [ ] Максимальный размер файла: 10 МБ
- [ ] После загрузки текст отображается в поле «Расшифровка» карточки
- [ ] Исходный файл сохраняется в Supabase Storage и доступен для скачивания
- [ ] При ошибке чтения файла — понятное сообщение об ошибке

---

### US-4: Комментарий Даниила (Даниил)

**Как** Даниил, **я хочу** оставить комментарий к встрече или профилю руководителя, **чтобы** он учитывался при подготовке следующей встречи.

**Acceptance Criteria:**
- [ ] Комментарий можно добавить на уровне профиля руководителя и на уровне карточки встречи
- [ ] Комментарии видны обоим пользователям
- [ ] Комментарии к последней встрече включаются в автоматически формируемый контекст следующей встречи
- [ ] Дата и автор комментария отображаются

---

### US-5: AI-генерация сценария (Саша)

**Как** Саша, **я хочу** нажать кнопку «Сгенерировать сценарий» в новой карточке встречи **и получить** готовый сценарий на основе контекста, **чтобы** не писать его с нуля.

**Acceptance Criteria:**
- [ ] Кнопка генерации доступна, когда в карточке есть поле «Контекст из предыдущих встреч»
- [ ] Генерация происходит через OpenRouter API с выбранной моделью
- [ ] Индикатор загрузки отображается во время генерации
- [ ] Сгенерированный текст вставляется в поле «Сценарий» и доступен для редактирования
- [ ] При ошибке API — понятное сообщение, возможность повторить

---

### US-6: Переключение AI-модели (Даниил)

**Как** Даниил, **я хочу** выбрать AI-модель в настройках, **чтобы** использовать разные модели для разных задач.

**Acceptance Criteria:**
- [ ] В настройках доступен список моделей OpenRouter
- [ ] Выбранная модель сохраняется и применяется к последующим генерациям
- [ ] Текущая активная модель отображается в интерфейсе

---

### US-7: Цепочка встреч (Даниил)

**Как** Даниил, **я хочу** видеть историю всех встреч с руководителем в хронологическом порядке, **чтобы** отслеживать динамику.

**Acceptance Criteria:**
- [ ] На карточке руководителя отображается хронологический список встреч
- [ ] Каждая встреча — раскрываемая карточка (accordion)
- [ ] Видны: номер встречи, дата, статус
- [ ] Прогресс между встречами виден визуально (индикатор или таймлайн)

---

### US-8: Список компаний (Даниил и Саша)

**Как** пользователь, **я хочу** видеть все компании на главном экране с ключевыми метриками, **чтобы** быстро перейти к нужной.

**Acceptance Criteria:**
- [ ] Отображаются: название, количество активных руководителей, дата последней встречи, статус
- [ ] Статус компании: активна / завершена
- [ ] Клик по компании переходит на экран руководителей

---

### US-9: Список руководителей (Даниил и Саша)

**Как** пользователь, **я хочу** видеть всех руководителей компании с их статусом, **чтобы** быстро перейти к нужному.

**Acceptance Criteria:**
- [ ] Отображаются: имя, должность, тип работы, количество встреч, дата последней встречи, статус
- [ ] Клик по руководителю переходит на его карточку

---

### US-10: Смена статуса карточки встречи (Саша / Даниил)

**Как** Саша, **я хочу** менять статус карточки встречи по мере её прохождения, **чтобы** отражать реальное состояние.

**Acceptance Criteria:**
- [ ] Статусы: `подготовка → проведена → обработана → закрыта`
- [ ] Переход к следующему статусу доступен кнопкой
- [ ] Переход к «закрыта» доступен только если заполнены обязательные поля (выводы, план действий)
- [ ] Откат статуса назад не разрешён (статус только вперёд)
- [ ] Текущий статус визуально выделен

---

### US-11: Редактирование профиля руководителя (Саша)

**Как** Саша, **я хочу** заполнить профиль руководителя один раз, **чтобы** он был доступен для всех встреч.

**Acceptance Criteria:**
- [ ] Поля профиля редактируются в интерфейсе
- [ ] Изменения сохраняются автоматически (autosave) или по кнопке
- [ ] Все поля: имя, должность, роль, контекст, запрос от директора, сильные/слабые стороны

---

### US-12: Вход в систему

**Как** пользователь, **я хочу** войти через email и пароль, **чтобы** получить доступ к приложению.

**Acceptance Criteria:**
- [ ] Форма логина с полями email и пароль
- [ ] Неверные данные — понятная ошибка
- [ ] После входа — редирект на главный экран (список компаний)
- [ ] Сессия сохраняется (не нужно логиниться каждый раз)
- [ ] Выход из системы доступен

## 8. Ограничения и допущения

### Ограничения
- Приложение рассчитано на 2 фиксированных пользователей, не более
- MVP поддерживает только тип встреч «one-to-one / менторинг»
- Расшифровки вводятся вручную (загрузка файла), без автоматической интеграции с Plaud
- Приложение desktop-first, мобильная адаптивность базовая

### Допущения
- Оба пользователя имеют стабильный интернет
- Расшифровки из Plaud экспортируются в txt или docx
- Объём данных не превысит 500 карточек встреч в обозримой перспективе
- OpenRouter API доступен и работает стабильно

## 9. Глоссарий

| Термин | Определение |
|--------|-------------|
| Замкнутый цикл | Механизм автоматического переноса выхода встречи во вход следующей |
| Карточка встречи | Структурированный документ с данными по одной встрече |
| Контекст | Сжатая информация из предыдущих встреч, используемая при подготовке новой |
| Сценарий | План проведения встречи — вопросы, темы, цели |
| Промпт для расшифровки | Инструкция для AI, формирующая структурированный протокол из сырой расшифровки |
| Plaud | Устройство/сервис для записи и расшифровки встреч |
| OpenRouter | API-агрегатор для работы с различными LLM-моделями |
